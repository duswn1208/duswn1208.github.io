Project 01

Redis 기반 중복 발송 방지 공통 라이브러리 설계

Redis SETNX 기반 분산 락과 수신자 샘플링 정규화 구조를 표준화한 공통 모듈.

해결

  • 사용자 고유 키 · 발송 유형 · 메시지 본문 데이터를 조합한 해시 생성 로직 구현
  • 대규모 수신자 발송 환경 부하 방지를 위한 클라이언트 정의형 수신자 샘플링 구조 설계
  • Redis SETNX + TTL(60분) 분산 락으로 진입 단계에서 중복 요청 원천 차단, 발송 퍼블리싱 실패 시 캐싱 키 즉시 삭제 롤백 전략 수립

성과

  • 중복 발송 관련 CS 유입 원천 방지 · 공통 라이브러리화로 팀 내 5개 서비스 적용 비용 60% 절감
회고 공통 모듈이라도 도입 비용이 높으면 다른 팀에서 안 쓰게 된다고 생각합니다. 이 프로젝트 이후로 인터페이스 단순화를 처음부터 신경 쓰게 되었습니다.
Project 02

Redis Sentinel Failover 시 Lettuce 커넥션 장애 대응

네트워크 순단 시 Lettuce 커넥션 스태킹으로 웹 서버 스레드가 마비되는 장애를 자동 복구하도록 개선.

문제

  • Sentinel Failover 후에도 Lettuce가 구(舊) 마스터 IP로 끊임없이 재연결 시도 → 명령 큐 적재(커넥션 스태킹) → Tomcat 스레드 풀 마비로 2차 장애 전파

분석

  • kill -9는 소켓 폐기로 재현 불가 → kill -STOP으로 Gray Failure(좀비 커넥션) 상태를 정교하게 유도 · 재현 성공
  • 단순 타임아웃 · 리트라이 · 풀링 옵션 조정만으로는 죽은 마스터에 매달린 좀비 커넥션 해소 불가 확인

해결

  • 자동 재연결은 유지하되, 단절 구간에는 명령을 버퍼링하지 않고 즉시 실패 처리(Fail-Fast)하도록 Lettuce 옵션 재구성 → 스레드 풀 마비 방어
  • 애플리케이션 레벨에서 PING 기반 연결 유효성 검증을 활성화 → OS 타임아웃을 기다리지 않고 Sentinel이 승격시킨 신규 마스터로 즉시 전환되도록 개선

성과

  • Failover 후 수동 개입 없이 수 초 내 서비스 자동 복구(Self-healing) 확인
  • kill -STOP 기반 장애 시뮬레이션 기법 정립 → 향후 인프라 리스크 대응 프로세스 자산화
회고 오픈소스 라이브러리의 기본 옵션이 우리 환경에 안 맞을 수 있다는 걸 확인한 프로젝트라고 생각합니다. 이 일을 계기로 새 라이브러리를 도입할 때 "이게 죽으면 우리 서비스는 어떻게 동작하나"부터 확인하는 습관이 생겼습니다.
Project 03

백그라운드 데이터 수집 병목 분석 및 MongoDB I/O 최적화

오픈 전 가상 서버의 CPU 90~100% 병목 원인을 추적하고, 데이터 수집 로직을 격리해 안정화한 프로젝트.

분석

  • CPU 90~100% 임계치 도달로 정상 테스트 불가 → MongoDB가 주원인임을 시스템 리소스 추적으로 확인
  • 조회 지연(2,000ms)을 인덱스로 해결 시도 → 오히려 4초 이상으로 악화 → 조회가 아닌 쓰기(Write) 트래픽에 의한 락(Lock) 병목임을 인지
  • 백그라운드 수집 프로그램 격리 테스트 → 센서 활성 여부 무관하게 Full-Table Write를 무한 루프로 수행하던 구조적 결함 규명

해결

  • 무거운 수집 로직을 메인 루프에서 제거 → 1시간 주기 독립 배치(Cron)로 격리 분리
  • 비활성 센서는 업데이트 대상에서 사전 필터링하는 방어 로직 추가 → 불필요한 디스크 I/O 원천 차단

성과

  • 서버 CPU 사용률 90% → 30% 안정화, 디스크 쓰기 22KB → 15KB 감축
회고 느린 조회를 인덱스로 해결하려다 더 악화된 경험이었습니다. 이 일 이후로는 증상이 아니라 원인을 먼저 보려고 노력하게 되었고, 가설을 세우고 격리 테스트로 좁혀가는 방식이 결국 가장 빠른 디버깅 경로라고 생각합니다.
Project 04

외부 AI 서비스 연동 중계 API 개발

사내 AI팀이 제공하는 문자 생성 · 추천 API 결과를 메인 웹 서비스에 결합하기 위한 RESTful 중계 API.

해결

  • 웹 클라이언트 파라미터(발송 목적 · 타겟층 레이블 등)를 외부 AI 연동 서버 통신 규격에 맞게 정제 → HTTP 통신 인터페이스로 라우팅 처리
  • 외부 AI 서버가 리턴한 JSON 응답을 안정적으로 파싱한 후 프론트엔드 렌더링 포맷으로 가공하여 중계 엔드포인트 구축

성과

  • 기존 레거시 아키텍처에 영향 없이 신규 외부 서비스 유연 · 신속 통합 완료
  • 외부 코어 레이어 ↔ 비즈니스 레이어 분리 구조에서 API 연동 · 클라이언트 대응 데이터 핸들링 역량 확보
회고 외부 서비스 연동은 본 서비스에 영향을 주지 않도록 레이어 분리가 중요하다고 생각합니다. 응답 검증과 명확한 인터페이스 계약을 미리 잡아두는 게 결국 통합 속도를 좌우하는 부분이었습니다.
